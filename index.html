Introduction

This project is meant for those who want to create an API with DDD principles in mind.
It is separated in layers and, within each layer, the responsibilities are grouped in folders.

Scope of the project

Even though this project just wants to serve as an example, it is important to provide with some context to understand what is trying to solve.

The project initially arose as a need to regulate compulsive gambling. Basically, it allows customers to create deposit limits and self-exclusions within a period of time.
For example, imagine that a customer wants to limit the amount of money he can deposit in his account to a maximum of 100USD per week. Or imagine that this customer directly prefers to not be able to login in his account losing therefore the chance to see what's going on in the page.
Or imagine that he doesn't even want to receive marketing campaigns in his mailbox.

You will agree with me that the ability for the customer to apply these rules definitely attenuates his impulses to gamble and promotes auto-control.

Project structure

The Api layer does all the work around resources.
If you look at the controllers, you will see that each controller represents a resource.
There, you can also see that each resource can support multiple Http Verbs (GET, POST, PUT, DELETE)

The Application layer does all the work around intents.
Intents are reflected as requests in the code. There could be as many requests as the application requires.

The Domain layer does all the work around entities.
As described in the onion architecture or any domain-centric solution, this is the layer where all the other layers start from.

The Infrastructure layer does the integration with infrastructure matters (databases, queues, files, etc).

Validation

Validation intrinsically affects two layers - The Application layer and the Domain layer.

Imagine that there are a total of two developers collaborating in the project. One is the domain expert and the other knows how the application must behave (this includes knowing the intents of the user along the application).

Developer A, while developing the Domain layer, needs to make sure that no business rule is violated.
If any rule is violated, a custom exception of the type ConflictException will be thrown.
Ultimately, this exception will be transformed by the API into a Http Error Code of type 409 Conflict

Developer B, while developing the Application layer, needs to make sure that requests are properly formed. This includes standard validations like making sure the properties are well formatted and satisfy an specific range. The goal is to never let entities from the next layer be in an invalid state.
FluentValidation enormously helps with these type of validations. Something that I like is that it allows the validation to happen even before the request reaches the controller.
In this case, the API transforms these validations into 400 Bad request

If you are wondering why 409 is preferred over 400 in the Domain, the reasoning is that 400 explicitly says in its description that the request is malformed and, therefore, should not be re-sent without modifications. However, 409 says that the conflict is intrinsic to the current state and, therefore, could go away. This leaves us with the following basic Http Codes:

GET 200, 404, 500
POST 200, 400, 404, 409, 500
PUT 204, 400, 404, 409, 500
DELETE 204, 400, 404, 409, 500
